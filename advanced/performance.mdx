## Performance Overview

Flashlight is designed for speed, using a hybrid architecture that combines SQLite persistence with in-memory caching and parallel search algorithms.

### Key Metrics

| Metric | Performance |
|--------|-------------|
| **Initial Indexing** | ~1 million files in &lt;30 seconds |
| **Startup Time** | 1-2 seconds to load 1M entries from SQLite into memory |
| **Search Latency** | &lt;25ms typical, &lt;100ms worst case for 1M+ files |
| **Memory Usage** | 200-400 MB for 1M file index |
| **Index Storage** | ~100 MB per 1M files on disk |

<Info>
Performance numbers measured on modern hardware (Apple Silicon M1/M2 or equivalent Intel/AMD processors with SSD storage).
</Info>

## Indexing Performance

### Initial Index Build

The first time Flashlight runs, it performs a full filesystem scan:

```
Files Indexed    Time
─────────────────────────────
100,000         ~3 seconds
500,000         ~15 seconds
1,000,000       ~30 seconds
5,000,000       ~2.5 minutes
```

**Factors affecting indexing speed:**
- **Disk I/O speed**: SSDs provide 5-10x faster indexing than HDDs
- **Directory structure**: Deeply nested directories take longer to traverse
- **Excluded directories**: More exclusions = faster indexing
- **Number of files vs directories**: Directory traversal adds overhead

### Incremental Updates

After the initial index, subsequent startups are nearly instant because Flashlight:

1. Loads the index from SQLite (1-2 seconds for 1M entries)
2. Does NOT re-scan the filesystem
3. Relies on the file watcher for real-time updates

<Note>
No reindexing is required unless you explicitly trigger it or change indexed directories.
</Note>

### Windows MFT Optimization

On Windows, when running with administrator privileges, Flashlight can use the NTFS Master File Table (MFT) for **significantly faster** initial indexing:

- **Standard traversal**: ~30 seconds for 1M files
- **MFT enumeration**: ~10-15 seconds for 1M files (2-3x faster)

The MFT provides direct access to all files on an NTFS volume without traversing the directory tree.

## Search Performance

### Search Latency Breakdown

```
Query Type              Typical Latency (1M files)
────────────────────────────────────────────────────
Simple term            &lt;10ms
Multi-term             10-20ms
With operators         15-25ms
Wildcard pattern       20-40ms
Complex multi-query    30-100ms
```

### Parallel Search Algorithm

Flashlight uses the [rayon](https://github.com/rayon-rs/rayon) library for parallel searching:

```rust
let results = entries
    .par_iter()  // Parallel iterator across all CPU cores
    .filter(|entry| matches_query(entry, query))
    .map(|entry| (entry, calculate_score(entry, query)))
    .collect();
```

**CPU utilization:**
- Automatically detects available CPU cores
- Distributes search work across all cores
- Typical CPU usage: 50-80% across all cores during search
- Search duration: 10-50ms (most of which is parallel computation)

<Info>
On an 8-core processor, searches are typically 6-7x faster than single-threaded implementations.
</Info>

### Search Optimizations

#### 1. In-Memory Cache

All file entries are loaded into RAM at startup:

```rust
pub struct EntryCache {
    entries: RwLock<Vec<FileEntry>>,
    is_loading: AtomicBool,
}
```

- **Zero disk I/O** during searches
- Instant access to all file metadata
- Memory usage: ~300-400 bytes per file entry

#### 2. ASCII Fast Path

For queries without diacritics (most common case), Flashlight uses optimized ASCII comparison:

```rust
fn is_ascii_only(s: &str) -> bool {
    s.bytes().all(|b| b < 128)
}
```

This avoids expensive Unicode normalization when not needed.

#### 3. Early Termination

Search stops collecting results after reaching the limit (default: 50):

```rust
if results.len() >= limit {
    break;  // Stop searching once we have enough results
}
```

#### 4. Extension Pre-filtering

Extensions are pre-lowercased and stored during indexing to avoid repeated lowercasing:

```rust
fn matches_extensions_prelower(name: &str, extensions_lower: &[String]) -> bool {
    // Extensions already lowercased, single comparison per file
}
```

## Memory Usage

### Breakdown for 1 Million Files

```
Component                Memory Usage
─────────────────────────────────────────
File entries (in-memory)  ~300-400 MB
Search scoring buffers    ~10-20 MB
SQLite connection pool    ~5-10 MB
Rust runtime overhead     ~20-30 MB
─────────────────────────────────────────
Total                     ~350-460 MB
```

### Memory Efficiency Optimizations

1. **Removed `name_lower` column**: Saves ~40 MB for 1M files by performing case-insensitive search at query time instead of storing lowercase names

2. **Shared parent paths**: Uses `Arc<str>` to deduplicate parent directory paths in memory

```rust
pub struct FileEntry {
    pub parent: Arc<str>,  // Shared reference to parent path
    pub name: Box<str>,    // Owned name
    // ...
}
```

3. **Compact integer timestamps**: Stores timestamps as `u32` (seconds since epoch) instead of `i64` for 50% memory savings

<Note>
For typical home directories (100k-500k files), memory usage is 50-200 MB, well within reasonable limits for modern systems.
</Note>

## Storage Performance

### SQLite Database

Flashlight uses SQLite with Write-Ahead Logging (WAL) mode for optimal performance:

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA cache_size = -64000;  -- 64 MB cache
```

**Benefits of WAL mode:**
- Concurrent reads during writes
- Faster write performance
- Better crash recovery

### Index Size on Disk

```
Files           Database Size
──────────────────────────────
100,000        ~10 MB
500,000        ~50 MB
1,000,000      ~100 MB
5,000,000      ~500 MB
```

The database stores:
- File paths, names, and metadata
- Indexed directory list
- Recent files history
- Search history

## File Watcher Performance

### Real-Time Updates

The file watcher monitors filesystem changes using platform-native APIs:

- **macOS**: FSEvents
- **Linux**: inotify
- **Windows**: ReadDirectoryChangesW (or USN Journal with elevation)

**Latency:**
- File creation/deletion detected: &lt;100ms
- Index updated: &lt;10ms
- Total time from change to searchable: &lt;110ms

### Windows USN Journal

On Windows with administrator privileges, Flashlight can use the USN (Update Sequence Number) Journal:

- **Standard watcher**: Recursive directory monitoring (higher overhead)
- **USN Journal**: Volume-level change notifications (more efficient)

<Info>
The USN Journal reduces CPU usage for file watching by ~50% on large directory trees.
</Info>

## Optimization Tips

### For Large Indexes (5M+ files)

1. **Exclude unnecessary directories**: Build artifacts, dependencies, caches
2. **Increase results limit conservatively**: Higher limits = longer sort times
3. **Use specific search operators**: Narrow search space with `ext:`, `in:`, `size:`
4. **Enable MFT/USN on Windows**: Run with admin privileges for best performance

### For Low-Memory Systems

1. **Limit indexed directories**: Index only essential locations
2. **Exclude large directories**: Skip media libraries, backups, VMs
3. **Set max depth**: Prevent indexing deeply nested structures

### For Slower Disks (HDDs)

1. **Initial indexing takes longer**: Be patient during first run (~5-10x slower than SSD)
2. **Consider selective indexing**: Index only frequently accessed directories
3. **Disable file watcher**: Set `watchForChanges: false` if experiencing lag

<Warning>
Flashlight is optimized for SSD storage. Performance on HDDs will be noticeably slower, especially during initial indexing.
</Warning>
